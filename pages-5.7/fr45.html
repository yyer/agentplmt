Detection Rules
循环检测规则
LOOP-SIMPLE 简单循环 - 同工具同参数重复
强制高危
同一代理在 LOOP_GUARD_TTL_MS (300000ms) 窗口内对同一工具使用相同参数重复调用超过 LOOP_GUARD_ALLOW_COUNT (3) 次。ClawAegis Loop Guard 直接通过 sessionKey|runId|stableArgsKey 键值对计数检测。
writeeditapply_patchexecbashmemory_store
⏱ TTL: 300000ms
🔜 阈值: 3 次
🔒 动作: BLOCK_REASON_LOOP
📦 Key: path-based (read/write/edit)
// ClawAegis security-strategies.ts - Strategy #10 loop_guard if (count > LOOP_GUARD_ALLOW_COUNT) { return { ...resolveMatchedResult(mode, BLOCK_REASON_LOOP, { count }), emitResultLog: !isModeEnforced(mode), level: "warn" }; }
LOOP-CROSS-TOOL 跨工具循环 - write→read→write 模式
观察中危
代理在短时间内执行 write → read → write 的跨工具循环模式，即先写入文件、再读取同一文件、又修改同一文件，形成语义上的循环。此类跨工具循环需要结合 InstanceCommand 时间序列分析，当前为观察模式。
write→read→writeedit→read→editapply_patch→read→apply_patch
⏱ 窗口: 60s
🔜 阈值: 2 轮
👁 动作: 记录并告警
🔎 检测: 命令序列模式匹配
LOOP-PING-PONG 代理乒乓 - AgentA→AgentB→AgentA 往返
强制严重
两个或多个代理之间形成互相调用链，如 AgentA 调用 AgentB、AgentB 又回调 AgentA，形成乒乓式循环。通过调用链拓扑分析检测闭环路径。当检测到 AgentA → AgentB → AgentA 的闭环时立即触发阻断，避免无限递归消耗资源。
A→B→AA→B→C→AA→B→A→B
⏱ 窗口: 300s
🔜 阈值: 1 次闭环
🔒 动作: 阻断并告警
🔎 检测: 有向图环检测

Anomaly Detection Engine
异常协同模式检测引擎
Exfiltration Guard Dispatch Guard
异常检测引擎综合使用 ClawAegis 的 Exfiltration Guard（追踪 sourceSignals、transformSignals、sinkSignals 信号流）、 Dispatch Guard（4 模式组: openclaw-cli、destructive-path、destructive-intent、bypass-tool-call）以及 调用链拓扑分析，检测以下五类协同异常模式。每类模式包含检测逻辑、阈值配置、严重级别和自动化响应动作。
ANOMALY-FANOUT 扇出爆炸 - 1→N→N² 调用扩散
严重强制
单个代理在短时间内向大量下游代理发起并行调用，且下游代理各自继续扩散，形成指数级扇出。这通常由失控的任务分解或恶意代理的拒绝服务攻击引起，可导致系统资源耗尽。检测方式: 监控每个代理的出度（out-degree）和调用链树的总节点数增长率。
out-degree > 8tree-size > 50growth-rate > 10/min
🔎 检测: 单代理出度 + 树总节点数 + 增长率
🔜 阈值: 出度>8, 节点>50
⚠ 严重性: Critical
🔒 动作: 阻断新增调用 + 告警
// 检测逻辑伪代码 if (agent.outDegree > FANOUT_LIMIT || chainTreeNodeCount > TREE_SIZE_LIMIT) { blockNewCalls(agent.id, "fan-out-explosion"); emitAlert("ANOMALY-FANOUT", severity="critical"); }
ANOMALY-CONFLICT 冲突升级 - 矛盾命令发送至同一代理
高危强制
多个上游代理向同一目标代理发送相互矛盾的命令（如一个要求 start、另一个要求 stop），导致目标代理状态混乱或反复重启。通过 InstanceCommand 的 idempotency_key 和 command_type 冲突检测，结合 CurrentConfigRevisionID 版本不一致分析。
conflicting-commandsrevision-mismatchstatus-flapping
🔎 检测: 相同 InstanceID 的矛盾 command_type
🔜 阈值: 矛盾命令 >= 2 对
⚠ 严重性: High
🔒 动作: 暂停目标代理 + 人工审批
冲突检测示例: InstanceID=22 同时收到
[delta-9] command_type=restart, idempotency_key=ik-d9-rst
[epsilon-2] command_type=stop, idempotency_key=ik-e2-stp
→ 检测到 restart vs stop 矛盾对，触发 CONFLICT 告警
ANOMALY-DEADLOCK 死锁 - 代理间互相等待
高危强制
两个或多个代理形成循环等待依赖，如 AgentA 等待 AgentB 的返回、AgentB 又等待 AgentA 的返回，导致所有涉及代理永久阻塞。通过 InstanceCommand 的 status=running 但无 finished_at 的超时检测，结合等待图（wait-for graph）的环检测算法分析。
circular-waittimeout-no-finishwait-for-cycle
🔎 检测: Wait-for graph 环检测 + 超时
🔜 阈值: 环检测 1 次 + 超时 300s
⚠ 严重性: High
🔒 动作: 中断最年轻等待 + 告警
ANOMALY-STARVATION 资源饥饿 - 单代理消耗全部令牌
中危观察
某个代理持续发起大量工具调用或消耗大量 token，导致其他代理因配额不足而无法执行任务。结合 ClawManager 的 Quota 系统和 CostRecord 追踪，当单个代理的资源消耗占比超过阈值时触发告警。Exfiltration Guard 的信号追踪也可辅助识别异常资源消耗模式。
token-ratio > 80%call-rate > 30/minquota-exhausted
🔎 检测: 单代理 token 占比 + 调用频率
🔜 阈值: 占比>80%, 频率>30/min
⚠ 严重性: Medium
👁 动作: 记录 + 速率限制建议
ANOMALY-CASCADE 级联故障 - 错误沿调用链传播
严重强制
调用链中某个代理失败（InstanceCommand status=failed + error_message 非空），错误沿调用链向上游传播导致更多代理失败。结合 Exfiltration Guard 的 runtimeRiskFlags 和 Dispatch Guard 的 blocked 标记，追踪错误在调用树中的传播路径。当一条链中连续 2 个以上代理失败时触发级联故障告警。
chain-failures >= 2error-propagationruntimeRiskFlags accumulation
🔎 检测: 调用链连续失败节点 + runtimeRiskFlags
🔜 阈值: 连续失败 >= 2 节点
⚠ 严重性: Critical
🔒 动作: 隔离失败子树 + 降级
// Exfiltration Guard runtimeRiskFlags 追踪 const runtimeRiskFlags = [ "sensitive-source-transform-sink-chain", "encoded-secret-outbound-chain", "legacy-service-script-chain", "encoded-outbound-chain-risk", "suspicious-outbound-service-script" ];
