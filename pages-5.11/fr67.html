Audit Log Cryptographic Signing & Tamper Protection
审计日志加密签名与防篡改系统

FR-67 对关键审计日志进行加密、签名和不可抵赖保护，防止攻击者通过删除或篡改日志来掩盖入侵痕迹。系统采用三层防护架构：加密层确保日志在存储时不被泄露，签名层通过数字签名保证每条日志的完整性和来源可信，篡改检测层在读取时验证哈希指纹以发现任何篡改行为。ClawAegis 的 SelfIntegrityRecord 机制为日志完整性提供了指纹基线模型，KubeArmor 的 IMA（Integrity Measurement Architecture）通过 BPF LSM hook bprm_check_security 实现文件级完整性度量，KubeArmor PKI 体系提供完整的 CA 证书签发和密钥管理基础设施。

Three-Layer Log Protection Architecture
三层日志防护架构

审计日志的防篡改保护建立在三个互补的安全层之上：第一层是静态加密（Encryption at Rest），确保日志文件在磁盘上以加密形式存储，攻击者即使获得原始磁盘也无法读取明文日志；第二层是数字签名（Digital Signature），每条关键日志条目在写入时使用私钥签名，验证方可通过公钥确认日志来源和完整性；第三层是篡改检测（Tamper Detection），基于 ClawAegis SelfIntegrityRecord 的指纹模型，对日志文件计算 SHA-256 哈希并在读取时比对，任何篡改都会导致哈希不匹配从而被检测。

1
🔒 加密层 — Encryption at Rest

日志文件在持久化到磁盘前进行 AES-256-GCM 加密。加密密钥通过 KSecure SecLog 的结构化日志通道传递，确保日志从产生到存储全链路加密。KSecure SecLog 结构化格式包含 time、action、path、operation、severity 等字段，加密后仍保留时间索引以支持范围查询。

// KSecure SecLog 结构化日志格式 type SecLog { Time string // "2026-04-30 14:23:01.000000" LogType string // "Audit" | "Security" | "System" Action string // "block" | "audit" | "allow" HostName string // 主机标识 Source string // 事件源进程 Path string // 目标文件/资源路径 Operation string // "read" | "write" | "exec" | "connect" PID/PPID/UID uint32 // 进程上下文 Severity string // "critical" | "high" | "medium" | "low" Tags []string // 附加标签 Message string // 人类可读描述 }
2
✍️ 签名层 — Digital Signature

关键日志条目在写入时使用 KubeArmor PKI 体系签发的签名密钥进行数字签名。KubeArmor 的 CertLoader 实现了三种证书加载策略：SelfSignedCertLoader（自签名，适用于单节点）、ExternalCertLoader（外部CA，适用于企业PKI）、K8sCertLoader（Kubernetes Secret，适用于集群部署）。签名算法默认 RSA-SHA256，密钥长度 RSA-2048 用于签名密钥，CA 根密钥使用 RSA-4096。每条签名的日志包含签名字段和证书链，验证方可追溯到可信 CA 根证书。

// KubeArmor CertLoader 接口定义 type CertLoader interface { GetCertificateAndCaPool() (*tls.Certificate, *x509.CertPool, error) } // 三种实现： // 1. SelfSignedCertLoader - 自签名模式（CA + 签名密钥） // 2. ExternalCertLoader - 外部PKI模式（预置CA证书） // 3. K8sCertLoader - K8s Secret模式（从Secret读取） type DefaultKubeArmorCAConfig = CertConfig{ CN: "kubearmor", Organization: "kubearmor", IsCa: true, KeyUsage: KeyEncipherment | DigitalSignature | CertSign | CRLSign, }
3
🛡️ 篡改检测层 — Tamper Detection

篡改检测基于两层指纹体系：ClawAegis SelfIntegrityRecord 提供应用级文件完整性基线，对日志文件及其元数据计算 SHA-256 哈希并持久化到 self-integrity.json；KubeArmor IMA 提供内核级完整性度量，通过 BPF LSM hook bprm_check_security 在可执行文件加载时计算文件哈希，存入 BPF LRU HashMap（kubearmor_ima_hash_map，容量 10240 条目）。两者的指纹数据互补：SelfIntegrityRecord 关注日志文件的完整性和防护路径覆盖，IMA 关注系统可执行文件和关键二进制的运行时完整性。当任一层的哈希比对失败时，立即触发告警并冻结受影响的日志段。

// ClawAegis SelfIntegrityRecord 类型定义 type SelfIntegrityRecord = { pluginId: string, // "claw-aegis" stateDir: string, // 插件状态目录 rootDir?: string, // 根目录 rootRealPath?: string, // 解析后的真实路径 protectedRoots: string[], // 受保护的根路径列表 fingerprints: Record<string, string>, // 文件路径 → SHA-256 哈希 updatedAt: number, // 最后更新时间戳 } // KubeArmor IMA BPF 数据结构 struct ima_hash { u8 digest[32]; // SHA-256 摘要 }; // BPF Map: kubearmor_ima_hash_map (LRU_HASH, max=10240)

ClawAegis Self-Integrity Protection Model
ClawAegis 自完整性保护模型

ClawAegis 的 SelfIntegrityRecord 是日志防篡改的核心模型。它在插件启动时对 SELF_INTEGRITY_FILES 列表中定义的源文件（包括插件代码、配置文件和日志文件）计算 SHA-256 哈希，将指纹存入 self-integrity.json 持久化文件。运行时每次读取受保护文件前都会重新计算哈希并与存储的指纹比对，如果发现不匹配则说明文件已被篡改。该模型同时保护插件状态目录（包含日志存储路径）免受未授权修改。protectedPaths 配置扩展了保护范围，确保攻击者无法通过修改日志文件或其存储目录来销毁入侵证据。

KubeArmor IMA Integration
KubeArmor IMA 内核级完整性度量

KubeArmor 的 IMA（Integrity Measurement Architecture）集成通过 BPF LSM hook 在内核层面提供文件完整性度量。当可执行文件通过 execve 系统调用加载时，bprm_check_security hook 被触发，bpf_ima_file_hash() 函数计算目标文件的 SHA-256 哈希并存入 BPF LRU HashMap。这个机制确保了系统关键二进制文件（如 /bin/、/sbin/、/usr/bin/ 下的程序）的完整性可以在运行时被验证，与 ClawAegis 的应用级完整性保护形成互补。HashContext 包含 ProcessHash、ParentHash、ResourceHash 和 HashAlgo 字段，为每条安全事件提供完整的哈希上下文。

// IMA BPF LSM Hook SEC("lsm/bprm_check_security") int ImaBprmCheckSecurity(struct linux_binprm *bprm) { // 计算可执行文件哈希 u8 digest[32]; bpf_ima_file_hash(bprm->file, digest); // 存入 BPF Map u32 pid = bpf_get_current_pid_tgid(); bpf_map_update_elem(&kubearmor_ima_hash_map, &pid, &digest, BPF_ANY); return 0; }

Log Signing & Tamper Detection Rules
日志签名与篡改检测规则

签名规则定义了哪些日志条目必须进行数字签名、何时触发完整性验证以及检测到篡改时的响应策略。所有规则支持三种运行模式：enforce（强制执行，阻止未签名日志写入）、observe（观察模式，仅记录未签名事件）和 off（禁用）。

Critical 关键日志强制签名

以下类别的日志条目在写入时必须使用 KubeArmor PKI 签发的签名密钥进行数字签名，未签名的条目将被拒绝写入。签名包含完整的证书链，可追溯到可信 CA 根证书（CN=kubearmor, O=kubearmor, RSA-4096），实现不可抵赖保护。

Critical 篡改告警与证据保全

当哈希比对发现不匹配或签名验证失败时，立即触发篡改告警。系统在告警同时启动证据保全程序：冻结受影响的日志段、将当前内存中的日志快照写入只读证据区、通过 syslog 转发告警到外部 SIEM 系统。篡改响应策略支持三种模式：alert（仅告警）、freeze（冻结受影响日志段）、quarantine（隔离整个日志存储并切换到备份存储）。

IMA 通过 BPF LSM hook 在 execve 时计算文件哈希，BPF Map 容量 10240
列入此列表的文件将在启动时和周期验证时计算 SHA-256 哈希

⚠️ 篡改响应
freeze 模式冻结受影响日志段并保全证据；quarantine 模式隔离整个存储并切换备份
篡改检测时将内存中的日志快照和当前状态写入此只读路径
篡改告警通过 syslog 协议转发到外部 SIEM，确保告警不被本地销毁
支持 TCP 和 UDP 协议的 syslog 服务器

加密密钥与签名密钥分离存储，遵循最小权限原则
Acceptance Criteria
验收标准
✓ 关键日志有数字签名
安全事件、策略执行、防御事件等关键日志均使用 KubeArmor PKI 签发的密钥签名，包含完整证书链
✓ 日志篡改可检测
SelfIntegrityRecord SHA-256 指纹比对 + KubeArmor IMA 内核度量双重检测机制
✓ 日志有不可抵赖保护
每条签名日志包含 RSA-SHA256 签名 + 证书链 + 时间戳，签名者无法否认
