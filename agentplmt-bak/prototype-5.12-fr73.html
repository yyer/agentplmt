Confidential Computing Architecture
机密计算三层架构
✓ 已部署

机密计算架构为高敏Agent、高敏知识库、高敏数据处理链路提供使用态隔离保护。当前KubeArmor通过BPF-LSM/AppArmor/SELinux三后端强制提供软件级隔离基线，KSecure通过eBPF LSM Hook（inode_unlink/file_open/inode_rename）提供内核级隔离，TEE在此基础上叠加硬件级保护，实现从软件隔离到硬件隔离的纵深防御。ClawManager当前Instance模型缺少SecurityContext配置，需增强以支持TEE调度和机密工作负载。

当前隔离能力基线与增强路径
KubeArmor BPF-LSM 隔离基线 3后端
当前提供BPF-LSM/AppArmor/SELinux三后端强制隔离。DefaultPosture可将file/network/capabilities/device设为block实现强隔离。5个预设策略覆盖进程保护(protectProc)、环境隔离(protectEnv)等。IMA集成通过bprm_check_security LSM Hook和bpf_ima_file_hash()实现完整性验证。但当前仅软件级隔离，无法防御宿主机内核级窥探。
增强路径：DefaultPosture block模式 + TEE硬件隔离 = 软硬件纵深防御。TEE不可用时自动降级为block模式兜底。
KSecure eBPF 隔离基线 内核级
当前通过3个eBPF LSM Hook提供内核级文件隔离：inode_unlink阻止未授权删除、file_open阻止未授权读取、inode_rename阻止未授权重命名。dmidecode -s system-uuid提供硬件身份标识，为远程验证(Attestation)的硬件身份组件奠定基础。当前可防御应用层和部分内核层威胁，但无法防御物理内存读取。
增强路径：eBPF LSM Hook + TEE内存加密 = 内核隔离+内存加密双重保护。system-UUID纳入远程验证身份链。
ClawManager SecurityContext 增强 需增强

内存加密隔离 (TEE Memory Encryption) 硬件级
TEE Enclave内存页由CPU加密引擎(MEE/EPC)保护，宿主机OS和Hypervisor无法读取Enclave内存内容。Intel SGX使用Enclave Page Cache (EPC)存储加密内存页，MEE在内存总线上加密/解密；Intel TDX使用TD加密内存，整个VM内存受保护；AMD SEV-SNP使用内存加密引擎+反向映射表防止内存重放攻击。当Enclave内代码访问EPC页面时，CPU自动解密；宿主机DMA读取仅获取密文。EPC大小影响可并发保护的Enclave数量，SGX2支持动态EPC页面添加(ENCLS[EAUG])。
SGX EPC加密 TDX VM加密 SEV-SNP内存加密 MEE总线加密
远程验证流程 (Attestation Flow) 硬件+服务
远程验证确保TEE环境可信：1) Enclave内生成Report(含MRENCLAVE/MRSIGNER度量值)；2) Quoting Enclave将Report签名生成Quote；3) Quote提交至远程验证服务(IAS/DCAP for SGX, AMD KDS for SEV)；4) 验证服务检查Quote签名和度量值；5) 返回验证结果。KSecure dmidecode -s system-uuid作为硬件身份组件参与身份绑定，确保物理机器身份与TEE身份关联。验证通过后授予数据访问权限。
Report生成 Quote签名 DCAP验证 system-UUID绑定
数据封存 (Data Sealing) 持久化保护
Data Sealing将Enclave内数据加密持久化，密钥绑定TEE硬件身份(MRENCLAVE或MRSIGNER)，确保只有同一TEE环境(或同一签发者)可解封数据。SGX支持两种Sealing策略：MRENCLAVE绑定(同一Enclave代码可解封)和MRSIGNER绑定(同一签发者的不同版本Enclave可解封)。适用于高敏知识库向量数据、加密模型参数、会话密钥等持久化场景。KSecure inode_unlink Hook确保Sealed Data文件不被未授权删除。
MRENCLAVE绑定 MRSIGNER绑定 AES-GCM-256加密 KSecure inode_unlink保护
进程隔离 (Process Isolation) 软件+硬件
软件级进程隔离由KubeArmor protectProc预设策略实现：限制进程执行、阻止未授权进程创建、防止进程间信号注入。KSecure eBPF LSM Hook提供内核级进程文件保护：file_open Hook阻止未授权进程读取关键文件，inode_rename Hook防止关键进程二进制被替换。硬件级进程隔离由TEE Enclave边界实现：Enclave内进程与宿主机完全隔离，即使宿主机root也无法访问Enclave内存。SecurityContext RunAsNonRoot+DropAllCapabilities确保Pod进程最小权限。

KubeArmor Preset Policies
KubeArmor 5个预设策略（软件级隔离补充）
KubeArmor

KubeArmor提供5个预设安全策略，作为TEE硬件隔离的软件级补充。在TEE不可用或作为纵深防御层时，这些策略提供进程/文件/网络/能力/环境维度的强制隔离。

预设策略保护维度关键能力与TEE协同状态
protectEnv 环境变量 阻止未授权环境变量读取/修改，防止敏感信息泄露 保护Enclave外环境变量，防止宿主机窥探敏感配置 ENFORCE
protectProc 进程 限制进程执行路径，阻止未授权进程创建和信号注入 保护Enclave外进程，防止进程逃逸攻击 ENFORCE
protectFile 文件系统 限制文件读写权限，保护Sealed Data和配置文件 保护Enclave外Data Sealing文件，防止未授权读取 ENFORCE
protectNetwork 网络 限制网络连接，仅允许白名单地址通信 限制Enclave外网络出口，防止数据外泄 OBSERVE
protectCapabilities Linux能力 限制Linux能力集，阻止提权和特权操作 与SecurityContext DropAllCapabilities协同，纵深防提权 ENFORCE

ModelArmor Integration
ModelArmor模型级安全与TEE协同
KubeArmor

KubeArmor ModelArmor对pickle反序列化注入提供模型级安全防护，与TEE内存隔离形成纵深：ModelArmor在软件层阻止恶意pickle载荷执行，TEE在硬件层防止模型参数和推理结果被宿主机窥探。两者组合覆盖模型安全的两个关键维度——防注入(软件)和防泄露(硬件)。

ModelArmor - 防注入层 软件级
检测pickle反序列化中的恶意载荷（os.system/subprocess/Popen等），阻止模型文件中的代码注入攻击。在模型加载前扫描pickle字节码，识别和拦截恶意__reduce__方法。覆盖torch.load/torch.save等常用模型加载路径。
TEE内存隔离 - 防泄露层 硬件级
模型参数和推理结果在TEE加密内存中处理，宿主机无法读取模型权重和推理中间结果。即使攻击者获得宿主机root权限，也无法通过内存转储获取模型参数。Data Sealing保护持久化模型文件，MRENCLAVE绑定确保只有授权Enclave可加载。

Remote Attestation Flow
远程验证流程
1
Quote Generation (Quote生成)
TEE Enclave内生成REPORT(含MRENCLAVE/MRSIGNER度量值+Enclave属性)。Quoting Enclave(AESMD)将REPORT转换为QUOTE，使用硬件签名密钥( Provisioned Key )签名。Intel SGX通过DCAP(Direct Attestation)获取证书链，AMD SEV通过ARK/ASK证书链签名。KSecure dmidecode -s system-uuid作为硬件身份组件嵌入Quote元数据，关联物理机器身份。
REPORT生成 QUOTE签名 DCAP/ARK证书 system-UUID
2
Quote Verification (Quote验证)
远程验证服务接收QUOTE，验证证书链有效性( Intel根证书→中间证书→QE证书 )，检查QUOTE签名和TCB状态(TCB版本是否最新)。确认MRENCLAVE度量值匹配预期值(白名单比对)。验证TCB状态不包括已知的SGX安全咨询(SA)补丁缺失。验证超时设置30秒，失败自动重试2次。
证书链验证 TCB状态检查 MRENCLAVE白名单 SA补丁检查
3
Identity Binding (身份绑定)
验证通过后，将TEE身份(MRENCLAVE+MRSIGNER)与KSecure system-UUID硬件身份、ClawManager Instance ID绑定，形成三元组：{InstanceID, system-UUID, MRENCLAVE}。后续数据访问和API调用需携带此三元组作为身份凭证。身份绑定记录存入Attestation Registry，支持查询和审计。
三元组绑定 Attestation Registry 身份凭证
4
Access Grant (访问授权)
身份绑定确认后，向Enclave授予数据访问权限。高敏知识库向量数据、加密模型参数、会话密钥等通过安全通道传输至Enclave。访问授权有效期24小时(可配置)，到期需重新验证。授权期间如检测到TCB状态变化(如新安全补丁发布)，自动触发重新验证。验证失败时KubeArmor DefaultPosture block模式自动激活作为降级保护。
数据访问授权 24h有效期 TCB变更检测 降级保护

TEE类型配置
默认TEE类型
根据工作负载规模选择合适TEE类型。SGX适合密钥管理/特征加密等小型计算，TDX/SEV适合模型推理/RAG检索等大型工作负载。
EPC内存分配策略
动态分配推荐SGX2+，支持ENCLS[EAUG]运行时添加EPC页面。超额分配模式仅在EPC不足时使用，会有性能降级。
默认EPC大小 (MB)
SGX EPC通常128MB-512MB, TDX/SEV无此限制(使用VM内存)。单个SGX Enclave最大EPC由硬件决定。
远程验证配置
验证频率
高敏工作负载建议24小时以内验证周期。验证越频繁安全性越高，但开销也越大。
Quote验证服务
DCAP推荐生产环境使用，支持离线验证。IAS需互联网连接。SEV使用AMD ASK/ARK证书链。
验证超时 (秒)
Quote验证超时时间。超时后自动重试，最多重试2次。
