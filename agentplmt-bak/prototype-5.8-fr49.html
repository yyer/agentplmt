Verification Stages
验证管道各阶段详解
5 阶段
1
结构校验 (Structure Validation)
校验步骤链完整性：每个步骤必须有明确的输入/输出定义，步骤之间依赖关系无断裂，目标分解层次不超过最大深度。检查工具使用顺序是否符合声明的能力范围，确保 Agent 计划中引用的每个工具都在 OpenClaw 注册的 Skill 清单中。
step_chain_integrity tool_registration_check max_depth_limit input_output_match
2
逻辑一致性检查 (Logic Consistency Check)
验证步骤依赖排序的正确性（拓扑排序无环），资源边界检查（步骤的资源消耗不超过系统配额），前置条件满足验证。检测循环依赖和死锁可能。通过有向无环图 (DAG) 分析步骤间的依赖关系，确保不存在自相矛盾的逻辑。
dependency_ordering resource_bounds circular_dep_detection dag_analysis
3
风险评估 (Risk Assessment)
通过 ClawManager RiskRule 正则匹配引擎检测计划中的高风险动作。每条 RiskRule 定义 Pattern (regex)、Severity (low/medium/high)、Action (allow/route_secure_model/block/require_approval)。系统自动推导 HighestSeverity 和 HighestAction，当 require_approval 等同于 block 时，自动标记计划需要人工复核。
RiskRule:Pattern(regex) HighestSeverity HighestAction require_approval=block
4
模糊意图检测 (Ambiguous Intent Detection)
由 ClawAegis Prompt Guard 策略 PG-02 驱动，检测计划中的模糊目标和冲突目标。当 Agent 的计划描述含有多义性词汇、缺少量化指标、或同时包含相互矛盾的目标时，触发二次确认流程。PG-02 与 PG-01 (Prompt Injection Detection)、PG-03 (Context Manipulation)、PG-04 (Harmful Content Detection) 协同工作，覆盖意图层面的完整安全检测。
PG-02:Ambiguous PG-01:Injection PG-03:Manipulation PG-04:Harmful
5
决策输出 (Decision Output)
根据前 4 阶段校验结果输出最终决策：Approve（全部通过，计划可执行）、Flag（存在标记项，需记录但可执行）、Require Confirmation（模糊意图或高风险动作，需人工二次确认）、Reject（逻辑错误或安全违规，计划禁止执行）。InstanceCommand 的 17 种命令类型（含 quarantine_skill、stop_openclaw、handle_skill_risk、disable_skill 等）作为最终执行的载体。
approve flag require_confirmation reject InstanceCommand
ClawAegis Strategy Integration
ClawAegis 10 策略防护管道集成
10 strategies

计划验证管道深度集成 ClawAegis 10 策略防护管道，其中 Prompt Guard (order 5-6) 是 FR-49 的核心承载模块，负责意图校验与模糊意图二次确认。Dispatch Guard (order 7-8) 作为高风险动作检测器，Script Provenance Guard (order 4) 验证计划中引用的脚本来源可信性。

🛡️ Prompt Guard (Order 5-6) — 核心承载
✓
PG-01: Prompt Injection Detection
检测计划描述中是否被注入恶意 prompt，防止攻击者通过计划文本间接操控 Agent 行为。识别 "ignore previous instructions"、"you are now" 等注入模式。
✓
PG-02: Ambiguous Intent Detection
FR-49 核心策略。检测模糊目标（"尽可能优化"、"快速处理"）、冲突目标（同时要求"安全"和"效率最大化"）、缺少量化指标的步骤。触发二次确认流程，要求 Agent 或人工澄清意图。
✓
PG-03: Context Manipulation
检测计划是否试图通过上下文操控绕过安全检查，如插入误导性的上下文信息、伪装系统指令等。防止 Agent 被诱导生成看起来合理但实际有害的计划。
✓
PG-04: Harmful Content Detection
检测计划中是否包含有害内容（如数据窃取、破坏性操作），与 PG-01~03 模式共享脚本来源验证逻辑。Script Provenance Guard 使用相同模式验证脚本可信性。
⚖️ Dispatch Guard (Order 7-8) — 高风险动作检测
⚠
openclaw-cli 模式组
检测计划中是否有直接操作 OpenClaw CLI 的命令（如 openclaw skill uninstall、openclaw instance stop），这些操作可能绕过管理层面的安全控制。
⚠
destructive-path 模式组
检测破坏性路径操作模式：rm -rf /、mkfs、dd if=、shred 等。当计划步骤包含此类命令时自动标记为高风险。
⚠
destructive-intent 模式组
检测破坏性意图模式：drop table、truncate、DELETE FROM（无条件）、format 等。识别计划中隐含的破坏性意图。
⚠
bypass-tool-call 模式组
检测绕过工具调用模式：base64 -d | sh、xxd -r | bash、eval、exec 等混淆执行。防止 Agent 通过编码绕过安全检查执行计划步骤。
📦 Script Provenance Guard (Order 4)
验证计划中引用脚本的来源可信性，使用 PG-01~04 相同模式进行脚本源验证。确保 Agent 计划不会执行来源不明或被篡改的脚本。IsVerified=True 的脚本才允许在计划步骤中引用。
ClawManager RiskRule Matching
RiskRule 正则匹配与风险决策流程
AI Gateway

计划验证的 Stage 3（风险评估）通过 ClawManager AI Gateway 的 analyzeWithCompiledRules 方法对计划内容逐条匹配已编译的 RiskRule。匹配结果自动推导 HighestSeverity（low < medium < high）和 HighestAction（allow < route_secure_model < require_approval < block），其中 require_approval 等效于 block，触发人工复核。

Acceptance Criteria
验收标准
✓
AC-1: 执行计划有逻辑一致性校验
所有 Agent 输出的计划必须经过逻辑一致性校验（步骤依赖排序正确、无循环依赖、资源边界合规）。验证管道 Stage 2 (Logic Consistency Check) 通过 DAG 分析确保计划逻辑自洽，不通过的计划自动 Reject。
✓
AC-2: 高风险动作自动标记
验证管道 Stage 3 (Risk Assessment) 通过 ClawManager RiskRule 正则匹配自动检测高风险动作，HighestSeverity=high 的动作自动标记为 Flag 或 Require Confirmation。Dispatch Guard 4 模式组作为辅助检测器。InstanceCommand 的 quarantine_skill、stop_openclaw、handle_skill_risk、disable_skill 等命令类型自动归类为高风险。
✓
AC-3: 模糊意图触发二次确认
验证管道 Stage 4 (Ambiguous Intent Detection) 由 ClawAegis Prompt Guard PG-02 驱动，当检测到模糊目标、冲突目标或缺少量化指标时，自动触发二次确认流程，要求 Agent 或人工澄清意图后方可继续。

Verification Rules
四类验证规则
CRITICAL 逻辑一致性 (Logic Consistency)
Enforce
校验步骤依赖关系的正确性和完整性。确保步骤链可构成有向无环图 (DAG)，不存在循环依赖；前置步骤的输出满足后续步骤的输入要求；每个步骤引用的工具在 OpenClaw Skill 注册表中存在且可用。资源消耗预估不超过系统配额。
触发条件: 步骤依赖图存在环 / 输入输出不匹配 / 资源超限
阈值: 最大深度 8 层 / 最多 50 步 / 循环检测 0 容忍
通过动作: pass → 进入下一阶段
失败动作: block → Reject 计划 + 记录错误详情
circular_dependency input_output_mismatch resource_overrun unregistered_tool max_depth_exceeded
HIGH 边界一致性 (Boundary Consistency)
Enforce
确保计划中的所有动作都在授权范围内执行，不超出 Agent 的权限边界。检测特权升级尝试（如从普通用户操作升级到 sudo 操作），验证计划步骤不会访问受保护的路径或资源。与 ClawAegis Protected Path Guard (order 3) 和 Command Block (order 5) 协同工作。
触发条件: 操作超出 Agent 权限范围 / 访问受保护路径 / 特权升级
阈值: 权限级别匹配 ≥ required_level / Protected Path 列表校验
通过动作: pass → 进入风险评估
失败动作: block → Reject + 安全告警
privilege_escalation protected_path_access scope_violation unauthorized_resource protected_path_guard command_block
HIGH 风险合理性 (Risk Rationality)
Enforce
高风险动作必须有合理的风险收益比。验证计划中标记为高风险的步骤（如删除操作、系统重启、安全策略变更）是否具备充分的风险缓解措施和回滚方案。ClawManager RiskRule 匹配结果中 HighestSeverity=high 且 HighestAction=require_approval 的步骤需要额外提供风险缓解说明。
触发条件: RiskRule HighestSeverity=high / Dispatch Guard 模式触发
阈值: 高风险步骤必须有缓解方案 / 回滚计划 / 理由说明
通过动作: flag → 标记为高风险但允许执行
失败动作: require_confirmation → 需人工确认
high_risk_no_mitigation no_rollback_plan cost_benefit_unreasonable dispatch_guard_trigger require_approval
HIGH 模糊意图 (Ambiguous Intent)
Enforce
由 ClawAegis Prompt Guard PG-02 驱动，检测计划描述中的模糊目标（"尽可能优化"）、冲突目标（"最大化效率"与"确保安全"同时存在）、缺少量化指标（"处理数据"而非"处理 1000 条数据"）。触发二次确认流程，要求 Agent 或人工澄清意图。
触发条件: PG-02 检测模糊/冲突/无量化目标
阈值: 模糊度评分 ≥ 0.6 / 目标冲突数 ≥ 1
通过动作: pass → 意图清晰，继续执行
失败动作: require_confirmation → 暂停执行 + 二次确认
PG-02:ambiguous_goal PG-02:conflicting_objective PG-02:no_quantification PG-02:vague_action
Dispatch Guard Pattern Groups
Dispatch Guard 4 模式组 — 高风险动作检测器
ClawAegis Order 7-8

Dispatch Guard 的 4 个模式组在计划验证管道中作为高风险动作的辅助检测器，与 RiskRule 正则匹配互补。当 Dispatch Guard 检测到计划步骤匹配任一模式组时，该步骤自动标记为高风险并触发风险合理性检查。

CRITICAL openclaw-cli — CLI 直操检测
检测计划步骤中是否包含直接操作 OpenClaw CLI 的命令。Agent 应通过 API 层面操作，而非直接调用 CLI 命令，以避免绕过管理层面的安全控制和审计追踪。
openclaw skill uninstall openclaw instance stop openclaw config reset openclaw agent restart
CRITICAL destructive-path — 破坏性路径操作
检测文件系统破坏性操作，包括强制递归删除、磁盘格式化、数据销毁等。此类操作一旦执行不可逆，在计划中必须具备完整的回滚方案和风险缓解说明。
rm -rf / mkfs.ext4 dd if=/dev/zero shred -u wipefs
HIGH destructive-intent — 破坏性意图
检测数据库层面的破坏性操作和系统级配置变更。无条件删除、表销毁等操作在计划中出现时，需要 RiskRule 匹配确认并提供替代方案。
DROP TABLE TRUNCATE TABLE DELETE FROM (无条件) format partition
HIGH bypass-tool-call — 绕过工具调用
检测编码混淆和动态执行模式，Agent 通过 base64 解码、xxd 逆转换、eval/exec 动态执行等方式绕过安全检查。此类模式在计划中出现时自动标记为高风险并触发二次确认。
base64 -d | sh xxd -r | bash eval $VAR exec $(cmd) python -c "exec(...)"
